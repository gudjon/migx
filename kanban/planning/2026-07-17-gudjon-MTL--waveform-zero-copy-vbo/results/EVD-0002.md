# EVD-0002 — persistent VBO vs the no-VBO client-array path

Measures the effect of replacing the per-draw client-memory vertex bind
(`src/rendergraph/opengl/backend/basegeometrynode.cpp`) with a **persistent VBO** (upload once, re-upload
only on dirty, draw from GPU memory). Measured against `EVD-0001` (baseline dossier).

## Environment
Same host as EVD-0001: Apple **M4** (10-core), macOS 26.2, arm64 native, **RelWithDebInfo**,
Qt 6.8.3 (buildenv `mixxx-deps-2.6-arm64-osx-aa78b5a`). Fork HEAD `e099d24ac8` (working tree modified with
the change; not committed). Graphics backend OpenGL (`rendergraph_gl`, the default QWidget-skin path).

## The change (what shipped in the working tree)
| File | Change |
|---|---|
| `src/rendergraph/opengl/backend/basegeometrynode.h` | VBO state (`m_vboId`, `m_vboByteSize`, `m_geometryDirty`), `markGeometryDirty()`, destructor that frees the buffer only with a current GL context. |
| `src/rendergraph/opengl/backend/basegeometrynode.cpp` | `render()` now: lazily `glGenBuffers`; binds the VBO; re-uploads (`glBufferData` orphan + `glBufferSubData`, or reallocating `glBufferData` on size change) **only when dirty or size changed**; binds attributes with `setAttributeBuffer` (byte offset into the VBO) instead of `setAttributeArray` (client pointer); unbinds after draw. Destructor frees the buffer. |
| `src/rendergraph/opengl/geometrynode.cpp` | `markDirtyGeometry()` now sets the backend dirty flag (was a no-op) — the upstream renderers already call it on every geometry change. |
| `src/test/waveformrenderbenchmark.cpp` | Added `BM_WaveformVboUpload` (offscreen GL bench of the per-frame upload cost). |

**Correctness argument (why dirty-tracking is safe):** every renderer that mutates geometry already
calls `markDirtyGeometry()` (audited: 12/12 allshader renderers). This is the *same* `DirtyGeometry`
signal the **scenegraph** backend already requires for a geometry update to become visible
(`scenegraph/geometrynode.cpp` → `markDirty(QSGNode::DirtyGeometry)`). Honoring it in the GL backend
makes the two backends consistent — a renderer that mutated geometry without marking it dirty would
already be broken on the QML/scenegraph path today. The initial dirty flag is `true` and any vertex-count
change forces a reallocating upload, so the first frame and every resize upload unconditionally.

## Result 1 — CPU vertex rebuild is UNCHANGED (non-regression, `P-03`)
The backend change lives in `render()`, which `BM_Waveform` does not exercise (`preprocess()` never
touches a GL context). Confirms the change did not perturb the dominant CPU path.

`build/mixxx-test --benchmark --benchmark_filter=BM_Waveform`:

| Bench | p50 | p90 | p99 | min (floor) | EVD-0001 pinned |
|---|---|---|---|---|---|
| `BM_WaveformRGBPreprocess` | 39.3µs | 73.3µs | 122.2µs | 28.1µs | ~39µs floor / ~45µs p50 ✅ match |
| `BM_WaveformFilteredPreprocess` | 41.5µs | 60.6µs | 101.2µs | 32.3µs | ~51.5µs floor / ~60µs p50 (within load noise) |

→ CPU rebuild unchanged. ✅

## Result 2 — the eliminated per-frame upload (the win), measured
`BM_WaveformVboUpload` opens an offscreen GL surface and times one full upload of the reference-scene RGB
vertex buffer (**460 920 B ≈ 450 KB**, 23 046 vertices × 20 B): `glBufferData` orphan + `glBufferSubData`,
**no** `glFinish` in the timed region — i.e. the CPU-side, **render-thread occupancy** of one upload.
This is exactly the copy the new path does when a frame is dirty and **SKIPS (zero cost) when the
waveform is unchanged**.

`QT_QPA_PLATFORM=cocoa build/mixxx-test --benchmark --benchmark_filter=BM_WaveformVboUpload`:

| Run | p50 | p90 | p99 | min (floor) | max | bytes |
|---|---|---|---|---|---|---|
| Run 1 | 7.42µs | 12.6µs | 52.8µs | 6.58µs | 23446µs* | 460 920 |
| Run 2 | 7.08µs | 8.2µs | 45.3µs | 6.46µs | 8611µs* | 460 920 |

\* `max` (and the fat `p99`) are one-off warmup/scheduler/GPU-driver outliers (first-iteration buffer
allocation + host contention). The **floor (~6.5µs) and p50 (~7.1–7.4µs) are stable and reproducible**
across runs (~±5%). Use those (`P-03`/`P-18` — floor/p50, not the load-sensitive tail).

**Interpretation.** On the OLD (client-array) path, the driver marshaled this ~450 KB buffer CPU→GPU on
**every** `glDrawArrays`, every vsync, per deck — even for an unchanged waveform. The NEW (persistent-VBO)
path uploads only when the geometry is marked dirty; an **unchanged frame does zero upload** and draws
straight from GPU memory. So the render-thread cost eliminated per unchanged frame is **~6.5µs floor /
~7µs p50 per deck** at this scene, and it scales with pixel width and deck count. This ~7µs is a
**conservative floor** for the eliminated cost: it is the cost of an explicit, orphaned `glBufferSubData`;
the old implicit client-array marshaling at draw time also does per-attribute validation and is at least
as expensive.

## What is NOT measured (honest limits — `P-01`)
1. **Active scrubbing/playback** rebuilds + marks dirty every frame, so the VBO still uploads every
   frame there; the win in that regime is the (driver-dependent, likely modest but ≥0) difference between
   an explicit orphaned `DYNAMIC_DRAW` upload and the old implicit client-array copy — **not isolated
   here**. The measured, unambiguous win is the **idle/static-frame** case (paused deck, static window),
   which is extremely common and still redraws every vsync.
2. **End-to-end GPU frame time** (upload + shade + draw + composite) — needs the full widget + GUI event
   loop; **not measured**.
3. **Visual correctness** — cannot be seen headless. The bench and unit tests exercise the code path and
   pass, but the actual pixels must be eyeballed. **Flagged for human GUI verification before merge.**
4. `offscreen` QPA on macOS **cannot create a GL context** (`This plugin does not support
   createPlatformOpenGLContext`), so the bench SkipsWithError there; it was measured under `cocoa`
   (interactive session). Recorded honestly — not hidden.
