# EVD-0001 — M4 waveform render baseline

The pinned baseline every later MTL optimization measures its delta against (`P-25`).

## Environment (Wave 1 — captured)
| | |
|---|---|
| SoC | Apple **M4** — 10 cores (4 performance + 6 efficiency) |
| Arch | arm64 (native, `P-24` ✓ — `file build/mixxx` → arm64) |
| macOS | 26.2 |
| Build | **RelWithDebInfo** (optimized; the correct type for perf measurement) |
| Commit | `2332debead` (pinned) |
| Qt | **Qt 6.8.3** (buildenv `mixxx-deps-2.6-arm64-osx-aa78b5a`; confirmed from compile include paths `.../include/Qt6/QtCore/6.8.3`) |
| Graphics backend | **OpenGL (forced)** — `src/coreservices.cpp:826` `setGraphicsApi(OpenGL)`. Code reason: *"required to enforce ... OpenGL on all platforms to allow offscreen rendering to function as expected."* So the Metal switch is gated on solving offscreen-render-on-Metal. |
| Default renderer | `allshader::WaveformWidget` (RGB) on the `rendergraph_gl` backend (QWidget skins) |

## Method (Wave 2 — the benchmark)
A `benchmark::benchmark` case driving the waveform renderer over N frames of a scripted scrubbing
scene, reporting **p50 / p99 / max frame time + dropped-frame count** (`P-18`, not a mean). Built into
`mixxx-test`, run via `mixxx-test --benchmark --benchmark_filter=<Waveform...>`. Modeled on the
offscreen `QQuickRenderControl` harness in `src/test/controllerrenderingengine_test.cpp`. The hot path
under measure: `Engine::preprocess()` → per-frame vertex rebuild (`waveformrendererfiltered.cpp`) →
`basegeometrynode.cpp:83` client-side vertex bind → `glDrawArrays` (the no-VBO copy, the `P-22`/`AP-12`
target).

## Result (Wave 3 — Phase A captured, Phase B deferred)

**What was measured (Phase A — the dominant CPU-side per-frame cost).** A google-benchmark case,
`src/test/waveformrenderbenchmark.cpp` (wired into `mixxx-test` via the `BUILD_BENCH` source list in
`CMakeLists.txt`), drives the **real** allshader renderer's per-frame vertex rebuild —
`allshader::WaveformRenderer{RGB,Filtered}::preprocess()` → `preprocessInner()` — over a scripted
scrubbing scene. It builds a synthetic but realistically-sized `Waveform` (5-min track @ 44.1 kHz,
441 Hz visual rate → ~264 k `WaveformData` points), a dummy `Track`, and a `WaveformWidgetRenderer`
sized to a full-window Retina deck (1920×200 @ DPR 2.0 → 3840 px, default zoom 3.0). Each iteration
advances the play position and times one `preprocess()` call (the scalar per-pixel min/max reduction +
full client-side vertex-buffer rewrite — the no-VBO `P-22`/`AP-12` path). This is a **pure-CPU**
measurement: `preprocess()` touches only client-side geometry memory, needs no GL context, and adds
nothing to the RT audio path (`P-02` respected).

Reproduce: `build/mixxx-test --benchmark --benchmark_filter=BM_Waveform`
(4000 frames/iterationset; `->Iterations(4000)`, `kMicrosecond`, `UseRealTime`).

**Reproducibility (the baseline "threshold" per PS-MTL-01).** ⚠️ **Measured under heavy host load**
(`load avg ≈ 20` on 10 cores, 24 logged-in users — the M4 was ~2× oversubscribed during the run). The
consequence is spelled out honestly:
- The **uncontended floor (`min`) is highly reproducible**: RGB `38.6–39.2 µs` across 4 of 5 runs
  (one load-spike run hit 42.9); Filtered `51.4–51.8 µs`. Two-run delta on the floor ≈ **1–2 %**, well
  within a ±5 % tolerance. This is the trustworthy estimate of the true per-frame CPU rebuild cost.
- **`p50` is reproducible within ~±6 %** once the transient peak-load run (Run 2's 68 µs) is excluded:
  RGB `41.5 / 44.8 / 46.8 / 44.6 µs`.
- **`p99`/`max` are load-sensitive and NOT reproducible within tight tolerance** (RGB p99 `82–160 µs`,
  max `181–417 µs`) — they reflect scheduler contention, not the render path. A clean pinned tail number
  needs a re-run on a quiescent machine; flagged, not fabricated (`P-01`/`P-03`).

**Pinned baseline number (what later MTL optimizations measure against):**
> RGB default renderer, full-window Retina deck, default zoom — **per-frame CPU vertex rebuild
> ≈ 39 µs (floor) / ~45 µs (typical p50)**; Filtered ≈ **51.5 µs floor / ~60 µs p50**. Filtered costs
> ~1.3× RGB.

**Dropped frames.** Against a 60 Hz frame budget (16 667 µs) and 120 Hz (8 333 µs), the CPU vertex
rebuild alone dropped **0 / 4000 frames in every run** — even the worst single observed frame (850 µs)
is ~5 % of a 120 Hz budget. So at this scene the CPU preprocess is **~0.27 % of a 60 Hz frame / ~0.54 %
of 120 Hz** and is not itself the frame-drop source. Caveat (honest): this bench measures the CPU
rebuild only; a true end-to-end dropped-frame count requires the GPU render + full-buffer re-upload
(`glDrawArrays` over client memory) that this rebuild feeds — that is Phase B, not measured here. The
significance of the ~39 µs is that it is spent **every vsync, per deck, rewriting the entire vertex
buffer**, and it feeds a per-frame CPU→GPU copy (no VBOs) — the cost the zero-copy/Metal dossier targets.

**Phase B (offscreen GPU frame-time) — DEFERRED.** Not done. What it needs (for a follow-up wave):
an offscreen `QOpenGLContext` + `QOffscreenSurface` (+ `QQuickRenderControl` for the SG path), modeled
on `src/test/controllerrenderingengine_test.cpp`; call `rendergraph::Engine::initialize()` then drive
`Engine::preprocess()` + `Engine::render()` over N frames, timing the `basegeometrynode.cpp:83`
client-side vertex bind → `glDrawArrays` upload. Deferred deliberately (`confidence ≥ 0.4` fork rule):
Phase A already yields a real, pinned CPU baseline; forcing the GL harness now risked burning the wave
on offscreen-context plumbing. `status: partial` reflects this — Phase A complete, Phase B open.
