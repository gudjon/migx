# P-13 — Every current-state claim carries an inline verify command

## Statement
A doc/skill/AGENTS.md claim about the *current* state of the code — "X is wired to Y", "the flag is
off", "there's no direct Metal path" — carries an inline command that verifies it, and an agent
confirms it before acting on it. Trust the command's output, not the prose.

## Why
Current-state prose rots the moment the code moves; it's a snapshot, not a source of truth (MG-3).
An agent that trusts stale prose builds on a false premise. A claim that ships with its own
`grep`/`ctest`/`file` check is self-defending: the reader re-derives the fact from the code instead of
believing a sentence someone typed months ago. This is how the catalogue itself is meant to work — the
glossary derives the live ID list rather than restating it (`P-07`).

## How to apply
- When a doc states a current-state fact, append the command that checks it, e.g.
  `` (verify: !`grep -rl 'CMAKE_EXPORT_COMPILE_COMMANDS' CMakeLists.txt` ) ``.
- Before acting on such a claim, run the command; if output disagrees with the prose, the code wins —
  fix the doc.
- Prefer deriving the fact live (a generated section, an injected `` !`cmd` ``) over asserting it in
  prose at all (`P-07`).
- Reserve this for *current-state* claims; timeless doctrine (a pattern's rule) doesn't need a probe.

## Example — wrong
> AGENTS.md: "compile_commands.json is generated by the build." (It isn't today — a reader trusts it and
> wires clangd to a file that never appears; `AP-04`.)

## Example — right
> "No direct Metal path yet — rendergraph draws via Qt RHI (verify: `grep -rl MTLDevice src/` returns
> nothing)." The reader runs it and sees for themselves.

## Detection
Review: a current-state claim in docs with no verify command; an action taken on a doc assertion that
was never re-checked against the code.

## Cross-references
Serves `P-07` (derive, don't restate); its failure mode is `AP-04`. A correct compile database (`P-26`)
makes many such checks exact.
